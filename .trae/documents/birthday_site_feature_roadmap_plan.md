# 生日网站功能优化计划

## Summary

根据反馈，本次计划保留三个功能：

1. 移动端视口与底部安全区优化。
2. 首页测试入口弹窗升级。
3. 惊喜页摇一摇红包彩蛋。

前三个功能不做：

- 不做生日愿望胶囊。
- 不做生日拍立得相框。
- 不做最终生日长图。

本次实施目标是提高手机上的稳定性，把首页测试入口从浏览器原生 `prompt` / `alert` 改成更贴合页面风格的自定义弹窗，并把惊喜页最后的“扫码领红包”升级成“摇一摇领红包”。实现范围保持克制，不新增框架、不新增依赖、不改 GitHub Issues 逻辑、不改生日主流程、不改粒子和蛋糕动画。

另外补充一组“备选功能池”，仅供选择，不默认执行。用户如果选中其中某一项，再把它纳入执行范围。

## Current State Analysis

### 项目结构

当前项目主要由 4 个页面组成：

- `index.html`：生日倒计时、粒子开场、蛋糕、吹蜡烛、祝福文字、留言板、弹幕、主题切换。
- `photo.html`：摄像头拍照、重拍、保存、备注、照片时光轴、跳转惊喜页。
- `jingxi.html`：祝福弹窗、心形弹窗、烟花、最终祝福卡、保存图片、二维码彩蛋、留言板。
- `admin.html`：留言管理、照片管理、删除操作、登录入口。

当前 `README.md` 对页面结构的描述落后于实际项目，后续如果继续维护，应同步更新。

### 已有优势

`index.html` 已经形成完整主流程：

- 倒计时或 `?preview=1` 进入生日体验。
- 粒子开场结束后进入蛋糕。
- 用户吹蜡烛或手动完成愿望。
- 进入祝福文字。
- 跳转到 `photo.html`。

本次不改 `photo.html`、`admin.html`。`jingxi.html` 只改最后二维码红包彩蛋，不改烟花、留言板、保存祝福图等现有流程。

### 本次要解决的问题

第一，移动端仍有可优化空间。生日网站主要会在手机上打开，必须优先保证按钮不被安全区遮挡、弹窗输入框不被键盘挡住、全屏区域不因浏览器地址栏变化明显跳动。

第二，首页测试入口如果仍使用浏览器原生 `prompt` / `alert`，会破坏当前页面质感，也可能在自动化预览或部分浏览器环境中出现不支持问题。应改成页面内弹窗。

第三，惊喜页最后已有“扫码领红包”二维码弹窗，但当前是直接点击“惊喜”后弹出二维码。可以升级为更有互动感的“摇一摇领红包”：先提示用户摇一摇，摇完后金额从 1 元滚动到 6 元，再弹出二维码。

### 技术约束

必须遵守这些约束：

- 不引入 React、Vue、GSAP、Three.js 等框架。
- 不引入构建工具。
- 不新增外部依赖。
- 不破坏现有生日主流程。
- 不扩大公开 GitHub token 的写入场景。
- 新增动画继续支持 `prefers-reduced-motion` 降级。
- 优先手机适配。
- 每个功能应小步实施，便于回滚。

## Proposed Changes

### `index.html`

#### 功能一：测试入口弹窗升级

当前测试入口如果还使用浏览器原生 `prompt` / `alert`，建议替换成页面内小弹窗。

新增内容：

- 一个 `previewPasswordDialog`。
- 密码输入框。
- 错误提示文本。
- 确认和取消按钮。

修改位置：

- 替换测试入口按钮的点击逻辑。
- 保留原有密码判断和 `enterCelebration()`。

交互要求：

- 点击测试入口打开弹窗。
- 输入错误时在弹窗内显示“密码错误”。
- 支持 Enter 提交。
- 支持 Escape 关闭。
- 关闭后焦点回到测试入口按钮。

成功标准：

- 不再调用 `prompt()` 或 `alert()`。
- 浏览器控制台没有相关错误。
- 手机上输入密码不会遮挡确认按钮。

#### 功能二：移动端全屏和安全区优化

`index.html` 中所有关键全屏区域应兼容 `100dvh`：

```css
#countdownOverlay,
#introScreen,
#cakeScreen,
#blessScreen {
    min-height: 100vh;
    min-height: 100dvh;
}
```

底部按钮或靠近底部的操作区域应增加：

```css
padding-bottom: max(16px, env(safe-area-inset-bottom));
```

成功标准：

- 手机竖屏下吹蜡烛按钮、跳过按钮、祝福按钮不被底部遮挡。
- 微信内置浏览器或 Safari 地址栏变化时，页面不明显跳动。

### `jingxi.html`

#### 功能三：摇一摇红包彩蛋

当前惊喜页已有二维码弹窗：

```html
<div class="qrcode-overlay" id="qrcodeOverlay" aria-hidden="true" inert hidden>
    <div class="qrcode-modal" role="dialog" aria-modal="true" aria-labelledby="qrTitle" tabindex="-1">
        <img src="qrcode.png" alt="收款码" id="qrCodeImg">
        <div class="qrcode-title" id="qrTitle">🧋 请你喝奶茶</div>
        <div class="qrcode-sub">扫码领取 10 元红包 🧧</div>
        <button class="qrcode-close" id="qrClose" type="button">关闭</button>
    </div>
</div>
```

当前入口是：

```js
document.getElementById('nextBtn').addEventListener('click', openQrDialog);
```

计划改成：

- 点击 `nextBtn` 后不直接打开二维码。
- 先打开一个“摇一摇领红包”弹窗。
- 弹窗里提示“摇一摇，看看今天红包是多少”。
- 手机支持摇动时，摇动后触发金额滚动。
- 桌面或不支持设备运动权限时，提供一个按钮“点我摇一下”作为兜底。
- 金额数字从 `1` 滚动到 `6`，最终停在 `6 元`。
- 金额停住后显示“红包已就位”，再自动或点击打开二维码。
- 二维码弹窗文案从 `扫码领取 10 元红包` 改为 `扫码领取 6 元红包`。

新增 DOM：

```html
<div class="redpacket-overlay" id="redpacketOverlay" aria-hidden="true" inert hidden>
    <div class="redpacket-modal" role="dialog" aria-modal="true" aria-labelledby="redpacketTitle" tabindex="-1">
        <div class="redpacket-title" id="redpacketTitle">摇一摇领红包</div>
        <div class="redpacket-amount" id="redpacketAmount">¥1</div>
        <p class="redpacket-hint" id="redpacketHint">摇一摇，看看今天红包是多少</p>
        <button class="redpacket-shake-btn" id="redpacketShakeBtn" type="button">点我摇一下</button>
        <button class="redpacket-close" id="redpacketClose" type="button">关闭</button>
    </div>
</div>
```

新增 JS：

```js
const REDPACKET_FINAL_AMOUNT = 6;
let redpacketRolling = false;
let lastShakeAt = 0;

function openRedpacketDialog() {}
function closeRedpacketDialog() {}
function triggerRedpacketRoll() {}
function bindRedpacketShake() {}
function unbindRedpacketShake() {}
```

金额滚动逻辑：

- 使用 `setInterval` 或 `requestAnimationFrame` 在 `1` 到 `6` 之间快速切换。
- 滚动时间控制在短时间内，避免拖沓。
- 最终固定为 `6`。
- 滚动期间禁用按钮，防止重复触发。

摇一摇逻辑：

```js
window.addEventListener("devicemotion", onRedpacketMotion);
```

触发条件：

- 读取 `accelerationIncludingGravity`。
- 计算 x/y/z 加速度总和。
- 超过阈值触发。
- 设置冷却时间，避免连续触发。

兜底逻辑：

- iOS 或部分浏览器可能需要运动权限，也可能完全不触发 `devicemotion`。
- 因此 `redpacketShakeBtn` 必须始终可见。
- 用户点击按钮时和摇动手机执行同一个 `triggerRedpacketRoll()`。

和原二维码的衔接：

- 原 `openQrDialog()` 保留。
- `nextBtn` 的点击事件从 `openQrDialog` 改为 `openRedpacketDialog`。
- `triggerRedpacketRoll()` 完成后调用 `openQrDialog()`。
- 打开二维码前关闭红包弹窗，避免两个弹窗叠在一起。

成功标准：

- 点击最终祝福卡里的“惊喜”后，先出现摇一摇红包弹窗。
- 摇动手机能触发金额滚动。
- 点击“点我摇一下”也能触发金额滚动。
- 金额从 1 到 6 滚动，最终停在 6 元。
- 金额完成后弹出原二维码。
- 二维码文案显示 6 元红包。
- 关闭红包弹窗后焦点回到 `nextBtn`。
- 关闭二维码后焦点回到 `nextBtn` 或红包触发按钮。
- 不影响烟花、祝福卡、留言板和保存图片。

注意事项：

- 不要强依赖设备运动权限。
- 不要在弹窗关闭后继续监听 `devicemotion`。
- 不要使用浏览器 `alert()`。
- `prefers-reduced-motion` 下金额滚动可以缩短或直接淡入最终金额。

## Optional Feature Ideas

以下功能是备选项，不属于当前默认执行范围。它们都避开了已否定的三个方向：生日愿望胶囊、生日拍立得相框、最终生日长图。

### 备选一：生日盲盒抽签

功能概念：

用户点击一个“抽生日好运签”按钮，随机得到一张生日签，例如：

- 今日幸运关键词：进球
- 新一岁的隐藏属性：稳
- 生日好运指数：满格
- 专属祝福：愿你一路高光

推荐位置：

- `index.html`：祝福文字结束后，放在“进入拍照留念”附近。
- 或 `jingxi.html`：最终祝福卡中，作为一个轻量按钮。

实现方式：

- 新增一个本地数组 `luckyDraws`。
- 点击按钮后随机抽取一条。
- 抽签结果只在本地展示，不写入 GitHub。
- 可以用 `localStorage` 让同一次访问结果保持稳定。

优点：

- 低风险。
- 不需要网络。
- 手机点按体验好。
- 很适合作为第一批轻量增强。

注意：

- 不做复杂 3D 翻牌，避免手机卡顿。
- `prefers-reduced-motion` 下用淡入显示结果。

### 备选二：机长航线祝福小游戏

功能概念：

结合现有“机长”称呼，做一个轻量航线互动。用户点击几个航点，小飞机沿路线飞过去，每到一个航点解锁一句祝福。

示例航点：

- 起飞：生日快乐
- 巡航：一路顺风
- 穿云：心想事成
- 抵达：新岁高光

推荐位置：

- `jingxi.html`：烟花结束后的最终祝福卡上，增加按钮“开启生日航线”。

实现方式：

- 使用 DOM + CSS transform，不使用复杂 Canvas。
- 第一版不做自由拖拽，只做点击航点，降低移动端误触风险。
- 每个航点是一个按钮，点击后小飞机移动到下一个点。

优点：

- 专属感强。
- 和“机长”人设贴合。
- 不依赖网络和权限。

注意：

- 航点数量控制在 4 到 5 个。
- 减少动态效果下，小飞机直接跳到航点。

### 备选三：星光寻宝

功能概念：

在惊喜页或深色主题中放 5 颗可点击星星。用户点亮每颗星星后解锁一句短祝福，点满后出现隐藏祝福。

推荐位置：

- `jingxi.html`：烟花结束后，作为轻量彩蛋。

实现方式：

- 新增 `starBlessings` 数组。
- 生成 5 个可点击星星按钮。
- 点击后标记点亮状态，并显示对应祝福。
- 全部点亮后显示最终隐藏短句。

优点：

- 很轻量。
- 不依赖后端。
- 手机点击体验自然。
- 和星空、烟花、生日氛围一致。

注意：

- 可点星星要和装饰星星区分。
- 移动端点击区域至少 40px。

### 备选四：祝福弹幕剧场增强

功能概念：

现有首页已经有弹幕和留言读取能力，可以增加一个“播放祝福弹幕”按钮，让留言按节奏飘过，而不是无感地自动出现。

推荐位置：

- `index.html`：祝福文字区域或留言按钮旁。

实现方式：

- 新增显式按钮 `playBlessDanmaku`。
- 点击后从已有留言中选取短留言播放。
- 手机端同时最多显示 3 条。
- 留言加载失败时使用默认祝福。

优点：

- 复用现有留言数据。
- 不需要新数据结构。
- 比普通留言板更有仪式感。

注意：

- 不遮挡吹蜡烛按钮、跳过按钮、进入拍照按钮。
- 播放结束后清理定时器。

### 备选五：祝福反应墙

功能概念：

用户不想写留言时，可以点一个快速反应：

- 开心
- 好运
- 进球
- 干杯
- 生日快乐

页面展示每种反应累计数量。

推荐位置：

- `index.html`：留言按钮旁。
- `jingxi.html`：最终祝福卡下方。

实现方式：

第一版建议只用 `localStorage` 做本地计数，低风险。  
如果后续想做全站共享，再考虑写入 GitHub Issues 的 `reaction` 标签。

优点：

- 参与门槛低。
- 适合多人访问。
- 比留言更快。

注意：

- 如果写入 GitHub Issues，会增加公开 token 的写入频率，需谨慎。
- 需要限频，避免误点刷屏。

### 备选六：生日成就徽章

功能概念：

用户完成动作后点亮徽章：

- 看完开场
- 吹灭蜡烛
- 拍照留念
- 看完烟花
- 打开惊喜
- 留言成功

推荐位置：

- `jingxi.html`：最终祝福卡里显示。
- 或 `index.html`：祝福阶段显示一个小徽章入口。

实现方式：

- 用 `localStorage` 记录完成状态。
- 不写入后端。
- 未完成徽章显示为灰色，完成后点亮。

优点：

- 低风险。
- 有收集感。
- 适合作为最终祝福页的小彩蛋。

注意：

- 不要做成任务压力，只作为轻量纪念。

### 备选七：生日关键词云

功能概念：

读取留言后，按预设关键词统计常见祝福词，并生成漂浮词云。

关键词可以是：

- 快乐
- 好运
- 生日快乐
- 开心
- 高光
- 足球

推荐位置：

- `jingxi.html`：最终祝福卡下方。
- 或 `admin.html`：管理页统计区。

实现方式：

- 不做复杂中文分词。
- 只匹配预设关键词。
- 留言加载失败时不显示词云。

优点：

- 复用已有留言。
- 视觉轻量。
- 不新增写入。

注意：

- 留言数量少时效果可能一般。
- 需要默认空状态。

### 备选八：一键换主题氛围

功能概念：

在现有深浅主题之外，增加几套主题氛围：

- 蓝白清爽
- 黑白星空
- 奶油蛋糕
- 烟花夜空

推荐位置：

- `index.html`：主题按钮扩展为小菜单。

实现方式：

- 继续使用 CSS 变量。
- 不改页面结构。
- 保存到 `localStorage`。

优点：

- 可控、低风险。
- 能快速改变整体观感。

注意：

- 主题太多会让界面复杂。
- 第一版最多 3 到 4 个主题。

### 不建议优先做

以下方向不建议当前阶段做：

- 语音祝福上传：权限、兼容和文件体积风险高。
- 视频祝福：GitHub Issues 不适合存视频 Base64。
- 多人实时互动：当前静态站架构不适合实时通信。
- 复杂小游戏：容易拖慢手机，也偏离生日网站主体验。

## Assumptions & Decisions

- 该计划只规划三个已确认功能，不直接执行代码修改。
- 本次只改 `index.html` 和 `jingxi.html`。
- 不做生日愿望胶囊。
- 不做生日拍立得相框。
- 不做最终生日长图。
- 不修改 `photo.html`、`admin.html`。
- `jingxi.html` 只改最后红包二维码彩蛋，不改烟花、留言板、保存图片等流程。
- 不新增后端、不新增依赖、不新增框架。
- 不扩大 GitHub token 的使用范围。
- 不改生日主流程、粒子开场、蛋糕动画、拍照页、惊喜页主体流程。
- 测试入口弹窗必须支持键盘操作和关闭后焦点回归。
- 移动端适配只做布局稳定性增强，不改变视觉设计方向。
- 摇一摇红包必须有点击兜底，不能强依赖手机运动权限。
- 红包金额最终固定为 6 元，滚动过程展示 1 到 6 元的变化。

## Verification steps

### 首页流程验证

1. 打开 `index.html?preview=1`。
2. 等待粒子开场结束。
3. 进入蛋糕页。
4. 测试麦克风吹蜡烛成功路径。
5. 测试麦克风拒绝后的手动完成路径。
6. 验证祝福文字正常逐字显示。
7. 验证主题切换仍能同步背景和粒子颜色。
8. 验证测试入口弹窗不再调用 `prompt()` 或 `alert()`。
9. 验证错误密码只在弹窗内提示。
10. 验证正确密码仍调用现有 `enterCelebration()`。

### 惊喜页红包验证

1. 打开 `jingxi.html`。
2. 点击开启祝福，等待最终祝福卡出现。
3. 点击最终祝福卡中的“惊喜”按钮。
4. 验证先出现“摇一摇领红包”弹窗，而不是直接打开二维码。
5. 点击“点我摇一下”，验证金额从 1 到 6 滚动，最终停在 6 元。
